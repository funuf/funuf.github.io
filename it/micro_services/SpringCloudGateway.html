# 概述

Spring Cloud Gateway是Spring Cloud体系的第二代网关组件，目标是替代Netflix Zuul。

Spring Cloud Gateway基于Spring 5.0的新特性WebFlux进行开发，底层网络通信框架使用的是Netty，所以其吞吐量高、性能强劲。

Spring Cloud Gateway可以通过服务发现组件自动转发请求，默认集成了Ribbon做负载均衡，以及默认使用Hystrix对网关进行保护。

# 什么是服务网关

服务网关就是微服务系统的统一入口，进行服务的聚合的同时，隔离外围和核心机群。有以下好处：

  1. 对客户端而言，不需要了解各服务的具体部署方案，API的地址只需要配置Gateway的请求路径即可
  2. 对服务端而言，可以方便地进行流量监控、访问认证、访问统计、报文转换、跨域方案的单点实现

# 常用网关解决方案

# Nginx+Lua脚本

Nginx 是一个高性能的HTTP和反向代理服务器，通常作为门户/全局流量网关，处于最外层。

# Kong

Kong对Nginx+Lua进行了封装，提供了比Nginx更简单的配置方式，作为一款商业软件，还提供了一些优秀的插件。

# Traefik

作为知识储备要知道Traefik (opens new window) ，Traefik 是一个边缘路由器，这意味着它是您平台的大门，它拦截并路由每个传入的请求。

# Netflix Zuul

Zuul是Netflix公司开源的一个API网关组件。

Zuul1.x作为Spring Cloud Netflix的组件，本身基于单线程的接收请求和转发处理。

Zuul2.x基于Netty、非阻塞，支持长连接，但Spring Cloud暂时没有整合计划。

# Spring Cloud Gateway

Spring Cloud Gateway是Spring Cloud体系的第二代网关组件，目标是替代Netflix Zuul。

Spring Cloud Gateway旨在为微服务架构提供一种简单而有效的统一的API路由管理方式。

# Gateway特点

  • 基于Spring Framework5，Project Reactor和Spring Boot2.0
  • 易于编写的Predicates和Filter
  • 集成Spring Cloud DiscoveryClient
  • 路径重写
  • 限流
  • 动态路由（不需要知道upstream的IP，基于注册中心实现）

# 工作原理

如上图所示，客户端向Spring Cloud Gateway发出请求后，由Gateway Handler Mapping确定与请求相匹配的路由，然后将请求发送到Gateway Web Handler。

Gateway Web Handler通过指定的过滤器链将请求发送给实际的服务逻辑单元，然后返回。

过滤器由虚线分割的原因是，过滤器可以在发送请求之前和之后运行逻辑。过滤器的pre逻辑均被执行，然后发出代理请求，收到响应后，将运行过滤器的post逻辑。

# 核心概念

路由（Route）：路由是网关最基础的部分，路由信息由ID、一组断言、一组过滤器、目标微服务URI组成。如果断言为真，则请求被该路由转发给目标微服务。

断言（Predicate）：Java8中的断言函数。Spring Cloud Gateway中的断言函数输入类型是Spring5.0框架中的ServerWebExchange。允许开发者匹配来自Http Request中的任何信息，比如请求头和参数等。

过滤器（Filter）：标准的Spring Web Filter。Spring Cloud Gateway中的Filter分为两种，分别是Gateway Filter和Global Filter。过滤器会对请求和响应进行处理。

# 声明方式

Route所需的Predicate对象和Filter对象，由对应的PredicateFactory和FilterFactory负责创建。

声明Factory的时候由两种方式：简化声明和展开声明（复杂声明）。

# 简化声明

简化声明（Shortcut Configuration）由FilterName开头，后面跟着=，再后面跟着以逗号分割的参数列表

spring:
  cloud:
    gateway:
      routes:
      - id: after_route
        uri: https://example.org
        predicates:
        - Cookie=mycookie,mycookievalue

上述例子定义名为Cookie的Route Predicate Factory和该Factory所需的两个参数，cookieName=mycookie，cookieValue=myCookieValue

# 复杂声明

复杂声明（Fully Expanded Arguments）表明使用标准的yaml规范进行配置，如下所示。

spring:
  cloud:
    gateway:
      routes:
      - id: after_route
        uri: https://example.org
        predicates:
        - name: Cookie
          args:
            name: mycookie
            regexp: mycookievalue

# 谓词/断言

Spring Cloud Gateway创建Route对象时，使用RoutePredicateFactory创建Predicate对象，Predicate对象可以赋值给Route。

  • Spring Cloud Gateway包含许多内置的Route Predicate Factory
  • 不同的RoutePredicateFactory侧重于匹配Http Request的不同属性
  • 多个RoutePredicateFactory可以通过逻辑与（and）结合起来一起使用

# 过滤器

Spring Cloud Gateway的Filter可以分为GatewayFilter和GlobalFilter。

  • GatewayFilter：网关过滤器，需要在配置文件上进行配置。
    • 通过spring.cloud.routes.filters配置时，只作用在当前路由上
    • 通过spring.cloud.gateway.default-filters配置时，作用在所有路由上
  • GlobalFilter：全局过滤器，不需要在配置文件中配置，作用在所有的路由上。
    • 最终通过GatewayFilterAdapter包装称GatewayFilterChain

# 自定义GlobalFilter

全局过滤器（Global Filter）作用于所有的路由，不需要单独配置。可以通过它来实现很多统一化处理的需求，如权限认证，IP访问限制等

创建全局过滤器，只需要实现GlobalFilter、Ordered接口，并在类上增加@Component注解就可以使用过滤功能了。

@Component
public class AuthFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        final String token = exchange.getRequest().getQueryParams().getFirst("token");

        if (token == null || token.isEmpty()) {
            final ServerHttpResponse response = exchange.getResponse();

            // 封装错误信息
            final HashMap<Object, Object> responseData = Maps.newHashMap();
            responseData.put("code", 401);
            responseData.put("message", "非法请求");
            responseData.put("cause", "Tokenis empty");

            // 将信息转换为JSON
            final byte[] data = JSON.toJSONString(responseData).getBytes(StandardCharsets.UTF_8);

            // 输出错误信息到页面
            final DataBuffer buffer = response.bufferFactory().wrap(data);
            response.setStatusCode(HttpStatus.UNAUTHORIZED);
            response.getHeaders().add("Content-Type", "application/json;charset=UTF-8");
            return response.writeWith(Mono.just(buffer));
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return 1;
    }
}

/**
 * 实现 GatewayFilter 接口即可
 */
public class AuthGatewayFilter implements GatewayFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        final String token = exchange.getRequest().getQueryParams().getFirst("token");

        if (token == null || token.isEmpty()) {
            final ServerHttpResponse response = exchange.getResponse();

            // 封装错误信息
            final HashMap<Object, Object> responseData = Maps.newHashMap();
            responseData.put("code", 401);
            responseData.put("message", "非法请求");
            responseData.put("cause", "Tokenis empty");

            // 将信息转换为JSON
            final byte[] data = JSON.toJSONString(responseData).getBytes(StandardCharsets.UTF_8);

            // 输出错误信息到页面
            final DataBuffer buffer = response.bufferFactory().wrap(data);
            response.setStatusCode(HttpStatus.UNAUTHORIZED);
            response.getHeaders().add("Content-Type", "application/json;charset=UTF-8");
            return response.writeWith(Mono.just(buffer));
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return 1;
    }
}

@Configuration
public class GatewayFilterConfig {
    @Bean
    public RouteLocator routeLocator(RouteLocatorBuilder builder) {
        return builder
                .routes()
                .route(r ->
                        // 断言（判断条件）
                        r.path("/product/**")
                                // 目标URI，路由到微服务的地址
                                .uri("lb://product-service")
                                // 注册自定义GatewayFilter
                                .filters(new AuthGatewayFilter())
                                // 路由ID，唯一
                                .id("product-service")
                )
                .build();
    }
}

# 网关限流

限流，就是限制对请求的响应频率。

我们无法控制客户端行为（请求频率），但可以决定对请求的响应频率。

通过限流，可以很好地控制系统的QPS，从而达到保护系统的目的。

  • 用户增长过快（好事）
  • 某个热点事件（微博热搜）
  • 竞争对象爬虫
  • 恶意请求

# 计数器算法

基于变量实现，实际情况是，短时间内大量请求过来，起不到保护系统的作用。

# 漏桶算法

基于队列实现，能够起到保护系统的作用，但对于流量洪峰，其后一段时间服务将出现无法响应的假象。

# 令牌桶算法

基于队列实现，令牌桶中的令牌以固定速率产生，获得令牌的请求将会被响应。

即能够起到保护系统的作用，又可以保证服务响应的可持续。

/**
 * 限流规则配置类
 */
@Configuration
public class KeyResolverConfiguration {

    /**
     * 基于路径限流
     */
    @Bean
    public KeyResolver pathKeyResolver(){
        return exchange -> Mono.just(exchange.getRequest().getURI().getPath());
    }

    /**
     * 基于参数限流
     */
    @Bean
    public KeyResolver parameterKeyResolver(){
        return exchange -> Mono.just(exchange.getRequest().getQueryParams().getFirst("userId"));
    }

    /**
     * 基于IP限流
     */
    @Bean
    public KeyResolver ipKeyResolver(){
        return exchange -> Mono.just(exchange.getRequest().getRemoteAddress().getHostName());
    }
}

# 高可用网关

业内通常用多少个9来衡量网站的可用性，例如QQ的可用性是4个9，就是说QQ能够保证在一年里，服务在99.99%的事件是可用的，只有0.01%的事件是不可 用的，大约最多53分钟。

对于大多数应用，2个9是基本可用，3个9是叫高可用，4个9是拥有自动恢复能力的高可用。

实现高可用的主要手段是数据的冗余备份和服务的失效转移，网关的实现方案如下图所示。

一个请求过来，首先经过Nginx的一层负载，到达网关，然后由网关负载到真实后端，若后端有问题，网关会进行重试访问，多次访问后仍失败，可通过熔断或服务降级立即返回结果。

而且，由于是负载均衡，网关重试时不一定会访问到出错的服务。
