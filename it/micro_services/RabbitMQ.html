# 简介

消息 指的是两个应用间传递的数据。数据的类型有很多种形式，可能只包含文本字符串，也可能包含嵌入对象。

消息队列(Message Queue) 是在消息的传输过程中保存消息的容器。

在消息队列中，通常有生产者（发布者）和消费者（订阅者）两个角色。

生产者只负责发送数据到消息队列，谁从消息队列中取出数据处理，他不管。消费者只负责从消息队列中取出数据处理，他不管这是谁发送的数据。

使用消息队列的作用是：

  1. 异步通信
  2. 解耦
  3. 流量削峰

RabbitMQ是一款有broker的，轻topic的，由erlang语言开发，基于AMQP（Advanced Message Queue 高级消息队列协议）协议实现的消息队列，在分布式系统开发中应用非常广泛。

替代方案：

  1. ActiveMQ
  2. RabbitMQ
  3. Kafka
  4. RocketMQ

# 架构

相关概念说明如下：

  • Broker（Server）：消息队列服务进程，此进程包括两个部分：Exchange和Queue
  • Virtual Host：虚拟主机，用于进行逻辑隔离。一个virtual host可以有若干个Exchange和Queue，同一个virtual host中不能有想通名字的Exchange
  • Exchange：消息队列交换机，按一定的规则将消息路由转发到某个队列，对消息进行过滤
  • Queue：消息队列，存储消息的队列，消息到达队列并转发给指定的消费方
  • Bindings：Exchange和Queue之间的虚拟连接，binding中可以包括多个routing key
  • Routing key：是一个路由规则，虚拟机可以用它来确定如何路由一个特定消息
  • Producer：消息生产者，即生产方客户端，生产方客户端将消息发送到MQ
  • Consumer：消息消费者，即消费方客户端，接受MQ转发的消息

发送消息流程：

  1. 生产者和Broker建立TCP连接
  2. 生产者和Broker建立通道
  3. 生产者通过通道将消息发送给Broker，由Exchange将消息进行转发
  4. Exchange将消息转发到指定的Queue

接收消息流程：

  1. 消费者和Broker建立TCP连接
  2. 消费者和Broker建立通道
  3. 消费者监听指定的Queue
  4. 当有消息到达Queue时，Broker默认将消息推送给消费者
  5. 消费者接收到消息

# Java Client流程

使用Java Client的流程是：

  1. 声明交换机、声明队列、声明队列与交换机的绑定
  2. Producer在进行消息投递的时候，需要指定exchange和routingkey。前者可以为空（采用默认的direct的exchange），后者支持模糊（采用#和*进行通配）
  3. 消费者在进行消息监听的时候，只需要指定queueName即可。

# 消费模式（Queue-Consumer）

根据Queue和Consumer的对应关系的不同，RabbitMQ提供了若干种消息消费模式：单一消费、轮询消费、公平消费。

# 单一消费模式

单一消费模式指：一个Queue只有一个Consumer，该Consumer消费Queue中的所有Message。

# Consumer

public class Consumer {
    public static void main(String[] args) throws Exception {
        Connection connection = RabbitMQUtil.createConnection();
        Channel channel = connection.createChannel();
        channel.queueDeclare(RabbitMQUtil.QUEUE_NAME, true, false, false, null);
        channel.basicConsume(RabbitMQUtil.QUEUE_NAME,
                new DeliverCallback() {
                    @Override
                    public void handle(String consumerTag, Delivery message) throws IOException {
                        System.out.println("【Consumer】DeliverCallback-------"+new String(message.getBody(), StandardCharsets.UTF_8));
                    }
                },
                new CancelCallback() {
                    @Override
                    public void handle(String consumerTag) throws IOException {
                        System.out.println("【Consumer】CancelCallback-------");
                    }
                }
        );
        // 阻塞，防止消费者JVM退出
        System.in.read();
    }
}

# Producer

import static fun.hellofun.rabbitmq_client.RabbitMQUtil.QUEUE_NAME;

public class Producer {

    public static void main(String[] args) throws Exception{
        // 1. 连接工厂声明RabbitMQ的server
        Connection connection = RabbitMQUtil.createConnection();

        // 2. 使用Connection中的一个Channel
        Channel channel = connection.createChannel();

        // 3. 声明队列，该操作可多次调用，队列不存在时创建，队列存在是忽略
        // 三参：如果我们声明一个独占队列（仅限于此连接），则为 true
        channel.queueDeclare(QUEUE_NAME,true,false,false,null);

        //4.发送消息给队列交换机  面试题：可以存在没有交换机的队列吗？不可能，虽然没有指定交换机但是一定会存在一个默认的交换机
        // 默认交换机是direct类型的，因此发送消息的时候要指定routingKey，否则消息不会被交换机投递到指定的Queue
        // 每一各Queue都有一个默认的 bindingKey，就是它的QueueName
        // 因此这里二参传递了QUEUE
        // Exchange和routingKey 不能为null 否则抛异常
        String message = "Hello";
        channel.basicPublish("",QUEUE_NAME, null,message.getBytes());

        //5.关闭通道，关闭连接
        channel.close();
        connection.close();
    }
}

# 轮询消费（Worker轮询均摊）

轮询消费是指：一个Queue有多个Consumer，RabbitMQ会采用轮询的策略，即不考虑各Consumer的实际处理能力，简单地将队列中的消息依次有序的发送给各个Consumer。

实现起来很简单，多个Consumer在调用channel.basicConsume的时候，使用同一个queueName即可。

如果队列中的一个message需要代表一个耗时任务，则可以指定多个消费者（worker）来消费同一个队列中的消息。此时Consumer更合适称之为Worker。

# 公平消费（Worker能者多劳）

在轮询消费的基础上，加上消息确认、持久化、声明Worker处理能力，即可实现公平消费

# 消息确认

RabbitMQ删除队列中的一个Queue的前提（不考虑queue超时和message超时）是：收到Consumer的ack信号。

确认信息有两种发送模式：自动消息确认和手动消息确认（类比MySQL的事务：自动事务和手动事务）

自动确认指的是，只要消费者收到消息，该Message就会从Queue中移除；而手动确认消息则需要开发着通过代码进行确认。

// 自动消息确认还是手动消息确认
boolean autoAck = true;
channel.basicConsume(queueName,autoAck,callback)

// 手动消息确认
// 二参：multiple true：确认多条消息 false：仅确认该表消息
channel.basicAck(messge.getEnvelope().getDeliveryTag(),false);

# 持久化

在声明队列和发送消息的时候，可以指定Queue和Message是否需要持久化，如果选择持久化，则RabbitMQ在重启后，未被消费的消息仍然得以存在。

// 声明队列时，指定Queue可以持久化
boolean durable = true;
channel.queueDeclare("task_queue",durable,false,false,null);

// 发送消息时，指定Message可以持久化
import com.rabbitmq.client.MessageProperties;

channel.basicPublish("","task_queue",MessageProperties.PERSISTENT_TEXT_PLAIN,message.getBytes());

# Worker处理能力

通过channel.basicQos声明Consumer的prefetch，即表明该Worker的处理能力。

# 公平消费

有了上面的基础，就可以声明公平消费模式了，即在未收到消息确认时，RabbitMQ不向此Consumer派发新的Message：通过channel.basicQos声明处理能力+手动消息确认进行实现。

channel.queueDeclare(TASK_QUEUE_NAME,true,false,false,null);
System.out.println(" [*] Waiting for message. To exit press Ctrl+C ");

// 在绑定消费者之前调用该方法
channel.basicQos(1);

DeliverCallback deliverCallback = (consumerTag,delivery) -> {
    String message = new String(delivery.getBody(),"UTF-8");
    
    System.out.println(" [] Received '"+message+"'");
    try{
      doWork(message);
    } finally {
        System.out.println("[x] Done");
        channel.basicAck(delivery.getEnvelope().getDeliveryTag(),false);
    }
}
channel.basicConsume(TASK_QUEUE_NAME,false,deliveryCallback,consumerTag->{});

# 分发模式（Exchange-->Queue）

在一次消息产生的场景下，Producer和Exchange是一对一的，因此需要找到三个需要分析的主要实体：Exchange、Queue、Consumer。

根据Exchange与Queue绑定关系不同，RabbitMQ提供了若干种消息分发模式：订阅分发、路由分发、主题分发（模糊路由）。

# 订阅模式（广播模式）

Exchange.type = fanout

一个Exchange绑定多个Queue，Exchange收到的Message将被投递到所有的Queue中。

在下面的样例代码中，Queue与Exchange绑定的时候，无需指定BindingKey；发送消息的时候，也无需指定routingKey。

# 交换机与队列

public class FanoutConstant {
    public static final String QUEUE_NAME_1 = "fanout_queue_1";
    public static final String QUEUE_NAME_2 = "fanout_queue_2";
    public static final String QUEUE_NAME_3 = "fanout_queue_3";
    public static final String QUEUE_NAME_4 = "fanout_queue_4";

    public static final String EXCHANGE_NAME = "fanout_exchange";
}

# 消费者

public class Consumer implements Runnable {

    @Override
    public void run() {
        final String queueName= Thread.currentThread().getName();
        System.out.println(queueName + "准备接受消息---------");
        try {
            Connection connection = RabbitMQUtil.createConnection();
            Channel channel = connection.createChannel();
            // 声明交换机
            channel.exchangeDeclare(FanoutConstant.EXCHANGE_NAME, BuiltinExchangeType.FANOUT,true,false,null);
            // 声明队列
            channel.queueDeclare(queueName, true, false, false, null);
            // 队列与交换机的binding关系
            channel.queueBind(queueName, FanoutConstant.EXCHANGE_NAME, "");
            channel.basicConsume(queueName,true,
                    new DeliverCallback() {
                        @Override
                        public void handle(String consumerTag, Delivery message) throws IOException {
                            System.out.println(queueName + "【Consumer】DeliverCallback-------" + new String(message.getBody(), StandardCharsets.UTF_8));
                        }
                    },
                    new CancelCallback() {
                        @Override
                        public void handle(String consumerTag) throws IOException {
                            System.out.println(queueName + "【Consumer】CancelCallback-------");
                        }
                    });
            System.in.read();
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    public static void main(String[] args) {
        new Thread(new Consumer(), FanoutConstant.QUEUE_NAME_1).start();
        new Thread(new Consumer(), FanoutConstant.QUEUE_NAME_2).start();
        new Thread(new Consumer(), FanoutConstant.QUEUE_NAME_3).start();
        new Thread(new Consumer(), FanoutConstant.QUEUE_NAME_4).start();
    }
}

# 路由模式（精确匹配）

Exchange.type=direct：routingKey与bindingKey精确匹配

Queue与Exchange绑定时，指定固定的bindingKey（也可以叫做routingKey）。

发送消息时也需要指定routingKey，Exchange会根据Message的routingKey，将其投递派发到具有等值的bindingKey的Queue中。

即routingKey与bindingKey的精确匹配。

在下面的样例代码中，Queue与Exchange绑定的时候，各BindingKey即各自deQueueName。

# 交换机与队列

public class DirectContant {

    public static final String QUEUE_NAME_1 = "direct_queue_1";
    public static final String QUEUE_NAME_2 = "direct_queue_2";
    public static final String QUEUE_NAME_3 = "direct_queue_3";
    public static final String QUEUE_NAME_4 = "direct_queue_4";

    public static final String EXCHANGE_NAME = "direct_exchange";
}

# 消费者

public class Consumer implements Runnable {

    @Override
    public void run() {
        final String queueName= Thread.currentThread().getName();
        String routingKey = queueName;
        System.out.println(queueName + "准备接受消息---------");
        try {
            Connection connection = RabbitMQUtil.createConnection();
            Channel channel = connection.createChannel();
            // 声明交换机
            channel.exchangeDeclare(DirectContant.EXCHANGE_NAME, BuiltinExchangeType.DIRECT,true,false,null);
            // 声明队列
            Map<String,Object> arguments = new HashMap<>();
            arguments.put("header1","value1");
            channel.queueDeclare(queueName, true, false, false, arguments);
            // 队列与交换机的binding关系
            channel.queueBind(queueName, DirectContant.EXCHANGE_NAME, routingKey);
            channel.basicConsume(queueName,true,
                    new DeliverCallback() {
                        @Override
                        public void handle(String consumerTag, Delivery message) throws IOException {
                            System.out.println(queueName + "【Consumer】DeliverCallback-------" + new String(message.getBody(), StandardCharsets.UTF_8));

                            channel.basicAck(message.getEnvelope().getDeliveryTag(),false);
                        }
                    },
                    new CancelCallback() {
                        @Override
                        public void handle(String consumerTag) throws IOException {
                            System.out.println(queueName + "【Consumer】CancelCallback-------");
                        }
                    });

            System.in.read();
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    public static void main(String[] args) {
        new Thread(new Consumer(), DirectContant.QUEUE_NAME_1).start();
        new Thread(new Consumer(), DirectContant.QUEUE_NAME_2).start();
        new Thread(new Consumer(), DirectContant.QUEUE_NAME_3).start();
        new Thread(new Consumer(), DirectContant.QUEUE_NAME_4).start();
    }
}

# 主题模式（模糊匹配）

Exchange.type=topic：routingKey与bindingKey模糊匹配

所谓主题模式，就是Exchange投递消息时需要满足Message的routingKey与Queue的bindingKey的模糊匹配。

当然，这需要routingKey和bindingKey满足特定的规则：

  1. 点分单词串 eg：quick.orange.raabbit
  2. *代表一个单词
  3. #代表任意个单词（0个或多个）
  4. routingKey不能使用通配符，它必须是明确的（即具有明确的业务）；而bindingKey可以是通配的。

# 交换机及队列

public class TopicConstant {
    public static final String EXCHANGE_NAME = "topic_exchange";

    public static final String QUEUE_1_NAME = "topic_queue_1_tel";
    public static final String QUEUE_2_NAME = "topic_queue_2_sms";
    public static final String QUEUE_3_NAME = "topic_queue_3_email";
    public static final String QUEUE_4_NAME = "topic_queue_4_ding";

    public static final String QUEUE_1_BINDING_KEY = "*.tel.*";
    public static final String QUEUE_2_BINDING_KEY = "*.sms.*";
    public static final String QUEUE_3_BINDING_KEY = "*.email.*";
    public static final String QUEUE_4_BINDING_KEY = "*.ding.*";
}

# 消费方

public class Consumer implements Runnable {
    private String queueName;
    private String queueBindingKey;
    public Consumer(String queueName, String queueBindingKey) {
        this.queueName = queueName;
        this.queueBindingKey = queueBindingKey;    }
    @Override
    public void run() {
        System.out.println("【Topic】模式" + queueName + "等待消息中...");
        try {
            Connection connection = RabbitMQUtil.createConnection();
            // api句柄
            Channel channel = connection.createChannel();
            // 声明队列
            channel.queueDeclare(queueName, true, false, false, null);
            // 声明交换机
            channel.exchangeDeclare(TopicConstant.EXCHANGE_NAME, BuiltinExchangeType.TOPIC);
            // 队列绑定
            channel.queueBind(queueName, TopicConstant.EXCHANGE_NAME, queueBindingKey);
            // 进行消费
            channel.basicConsume(queueName,
                    new DeliverCallback() {
                        @Override
                        public void handle(String consumerTag, Delivery message) throws IOException {
                            System.out.println(queueName + "【Topic】DeliverCallback---handle，消息是=" + new String(message.getBody()));
                        }
                    },
                    new CancelCallback() {
                        @Override
                        public void handle(String consumerTag) throws IOException {
                            System.out.println(queueName + "【Topic】CancelCallback---consumerTag=" + consumerTag);
                        }
                    });
            // 持续运行
            System.in.read();
        } catch (Exception e) {        } finally {        }
    }
    public static void main(String[] args) {
        new Thread(new Consumer(TopicConstant.QUEUE_1_NAME, TopicConstant.QUEUE_1_BINDING_KEY)).start();
        new Thread(new Consumer(TopicConstant.QUEUE_2_NAME, TopicConstant.QUEUE_2_BINDING_KEY)).start();
        new Thread(new Consumer(TopicConstant.QUEUE_3_NAME, TopicConstant.QUEUE_3_BINDING_KEY)).start();
        new Thread(new Consumer(TopicConstant.QUEUE_4_NAME, TopicConstant.QUEUE_4_BINDING_KEY)).start();
    }
}

Topic模式功能最为强大，可以以其他模式的行为运行：

  1. 当Queue的bindingKey是#的时候，它将收到所哟你消息，此时和fanout模式一样

# Header模式

Exchange.type=headers：非routingKey和bindingKey机制

通常，Exchange与Queue只能有一个bindingKey，且发送消息的时候只能指定一个routingKey，这通常是满足要求的。

但如果觉得还不够灵活（不能满足业务需求），则可以不实用routingKey和bindingKey这个机制。

RabbitMQ还支持Header模式。Header由headerName=headerValue格式构成：

  1. 为Queue打上若干个label（标签）
  2. Queue与Exchange绑定的时候，无需提供routingKey
  3. 发送消息的时候，提供对应的Headers
  4. 如果Headers与labels匹配，则Exchange会将消息投递到对应的Queue中
// 发送方
HashMap<String,Object> headers = new HashMap<>();
headers.put("labelKey","labelValue");
AMQP.BasicProperties properties = new AMQP.BasicProperties.Builder().headers(headers).build();
channel.basicPublish(DirectContaint.EXCHANGE_NAME,routingKey,properties,message.getBytes());

// 声明队列
Map<String,Object> arguments = new HashMap<>();
arguments.put("header1","value1");
channel.queueDeclare(queueName,true,false,false,arguments);

# RPC

尽管RPC是计算中非常常见的模式，但它经常收到批评。因为它导致系统不可预测，并给调试增加不必要的复杂性。滥用RPC不仅不会简化软件，还会导致代码无法维护。

准确来说，RPC模式不是消息投递模式（与fanout、direct、topic不再一个维度），只是通过RabbitMQ也可以实现RPC。

# Queue过期时间

在声明队列的时候，可以为该队列指定过期时间。队列中的消息，超过过期时间未被消费，就会被移除（如不移除到死信队列中，则会彻底丢失）。

@Configuration
public class RabbitMqConfig_Direct {
   @Bean
   public DirectExchange directExchange() {
      DirectExchange directExchange = new DirectExchange("direct_exchange", true, false);
      return directExchange;
   }

   @Bean
   public Queue directSmsQueue() {
      Map<String, Object> args = new HashMap<>();
      // 为队列指定过期时间，单位ms
      args.put("x-message-ttl", 5000);
      return new Queue("sms.direct.queue", true, false, false, args);
   }
}

# Message过期时间

在发送消息的时候，也可以为该消息指定过期时间。但与过期队列相比，有两处不同：

  1. 如果为Message指定过期时间，则在过期后会被直接移除，而非像过期队列那样，会被转移到私信队列中
  2. 如果同时为Message和Queue指定过期时间，则以最短的为准
@Test
void topicTest() {
  String routingKey = "sms";
  String playload = "张三";
  rabbitTemplate.convertAndSend("topic_exchange", routingKey, playload,
          new MessagePostProcessor() {
              @Override
              public Message postProcessMessage(Message message) throws AmqpException {
                  // 为Message设置过期时间
                  message.getMessageProperties().setExpiration("5000");
                  message.getMessageProperties().setContentEncoding("UTF-8");
                  return message;
              }
          });
}

# 死信队列

当消息在一个队列中变成死信（dead message）后，它能被重新发送到另一个交换机中，这个交换机就是DLX，与DLX进行绑定的队列就称为死信队列。

正常队列中的消息变成死信的原因如下：

  1. 消息被拒绝
  2. 消息过期
  3. 队列达到最大长度

DLX，全称为Dead-Letter-Exchange，可以称之为死信交换机，也有人称之为死信邮箱。DLX也是一个正常的交换机，和一般的交换机没有区别。

DLX能在任何的队列上被指定，实际上就是设置某一个队列的属性。当这个队列中存在死信时，RabbitMQ就会自动地将该死信重新发布到设置的DLX上去，进而被路由到另一个队列，即死信队列。

使用死信队列，只需要在定义队列的时候，设置队列参数x-dead-letter-exchange指定交换机即可。

@Configuration
public class RabbitMqConfig_Direct {
   @Bean
   public DirectExchange directExchange() {
      DirectExchange directExchange = new DirectExchange("direct_exchange", true, false);
      return directExchange;
   }

   @Bean
   public Queue directSmsQueue() {
      Map<String, Object> args = new HashMap<>();
      // 创建队列的时候，为其指定死信交换机
      args.put("x-dead-letter-exchange", "dead_fanout_exchnage");
      // 如果死信交换机的type不是fanout，则需要指定routingKey
      args.put("x-dead-letter-routing-key", "");
      return new Queue("sms.direct.queue", true, false, false, args);
   }
}

# 可靠生产

分布式事务是指一个事务内的操作被分散到多个节点（虚拟机节点/JVM节点）上，此时仍需要保证事务的ACID特性。

例如在下单场景下，库存和订单如果不再同一个节点上，就涉及到分布式事务问题，通过事务保证数据的完整性及一致性。

建议：

  1. 尽量去避免分布式事务问题（核心业务尽量不分）
  2. 尽量将非核心业务做成异步

# 可靠消费

一旦Message确认被Broker接收，则该Message将会存在于相应的Queue中。

而Queue中将一条Message删除的条件是：收到Consumer的ACK确认。默认有两种方式，一种是程序自动确认，另一种则是开发人员手动确认。

采用自动确认方式，如果消费者出现异常，则RabbitMQ会进行自动重发，最终会导致死循环。解决方案是：

  1. 控制重发次数
  2. try-catch+手动ack+死信队列处理+人工干预
